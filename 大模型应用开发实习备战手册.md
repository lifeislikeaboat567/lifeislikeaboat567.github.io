## 三、Transformer 与大模型基础

### 3.1 Transformer 整体架构

Transformer 由 **编码器（Encoder）** 和 **解码器（Decoder）** 堆叠组成，每层包含：
1. 多头自注意力层（Multi-Head Self-Attention）
2. 前馈全连接网络（FFN）
3. 残差连接 + 层归一化（Add & LayerNorm）

主流架构分类：

| 架构 | 代表模型 | 特点 | 典型应用 |
|-----|---------|-----|---------|
| Encoder-Only | BERT、RoBERTa | 双向注意力 | 文本分类、NER |
| Decoder-Only | GPT 系列、LLaMA | 因果注意力（单向） | 文本生成、对话 |
| Encoder-Decoder | T5、BART | 编解码交叉注意力 | 机器翻译、摘要 |

### 3.2 自注意力机制（Self-Attention）

**核心公式：**
```
Attention(Q, K, V) = softmax(QKᵀ / √dₖ) · V
```

其中：
- Q（Query）= X · Wq — 当前位置"要查询什么"
- K（Key）= X · Wₖ — 各位置"我能提供什么"
- V（Value）= X · Wᵥ — 各位置"实际内容"
- `√dₖ` 是缩放因子，防止点积值过大导致 softmax 梯度消失

**计算过程：**
1. 输入序列 X 通过三个线性变换得到 Q、K、V
2. Q 与 K 点积得到注意力分数矩阵
3. 除以 `√dₖ` 后做 softmax 归一化
4. 加权求和 V，得到每个位置的上下文感知表示

**相较 RNN/CNN 的优势：**
- 全局依赖建模：任意两个位置一步可达（RNN 需 O(n) 步）
- 完全并行化：训练速度快
- 动态权重：不同输入分配不同注意力

### 3.3 多头注意力（Multi-Head Attention）

```
MultiHead(Q, K, V) = Concat(head₁, ..., headₕ) · Wᴼ
headᵢ = Attention(Q·Wᵢᴼ, K·Wᵢₖ, V·Wᵢᵥ)
```

- h 组独立注意力并行运行，从不同子空间捕捉语义特征
- 最后将多头结果拼接，再经线性变换压缩回原始维度
- 典型：GPT-3 有 96 个头，每头维度 128

### 3.4 位置编码

Transformer 本身不感知位置顺序，需显式注入位置信息。

**正弦/余弦编码（原始 Transformer）：**
```
PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))
```
优点：无需训练，可外推到任意长度

**RoPE（旋转位置编码，LLaMA/GPT-NeoX 使用）：**
- 将绝对位置编码转化为对 Q/K 的旋转操作
- 天然编码相对位置关系，支持长度外推
- 是当前最主流的位置编码方案

**ALiBi（线性偏置注意力，MPT 使用）：**
- 在注意力分数上加线性偏置，距离越远惩罚越大
- 训练短序列可外推到长序列

### 3.5 层归一化（LayerNorm）

**公式：**
```
LayerNorm(x) = γ · (x - μ) / σ + β
```
- μ：均值；σ：标准差；γ、β：可学习参数
- 对每个 **样本** 的所有特征维度做归一化（区别于 BatchNorm 对批次归一化）
- 现代 LLM 多采用 **Pre-LayerNorm**（层归一化在 Sublayer 之前）以提升训练稳定性

### 3.6 GPT vs BERT 核心对比

| 维度 | GPT（Decoder-Only） | BERT（Encoder-Only） |
|-----|-------------------|---------------------|
| 注意力 | 因果掩码（只看左侧） | 双向（全局可见） |
| 预训练任务 | 语言建模（预测下一个 token） | MLM + NSP |
| 优势 | 文本生成、对话 | 理解任务、分类、NER |
| 代表应用 | ChatGPT、LLaMA | BERT 文本分类 |

## 四、大模型微调技术

### 4.1 微调方式总览

```
全参数微调（Full Fine-tuning）
  └── 所有参数参与更新，显存需求极高（通常需要 A100 集群）

PEFT（参数高效微调）
  ├── LoRA    — 低秩矩阵注入，训练 1-3% 参数
  ├── QLoRA   — LoRA + 4bit 量化，大幅降低显存
  ├── Adapter — 在每层插入小型瓶颈网络
  └── Prefix-Tuning — 可训练 soft prompt 前缀

对齐微调（Alignment Fine-tuning）
  ├── SFT（监督微调）— 基础对齐
  ├── RLHF（人类反馈强化学习）— PPO 对齐
  └── DPO（直接偏好优化）— 简化版 RLHF
```

### 4.2 SFT（监督微调）

**定义：** 在预训练大模型基础上，用高质量标注数据进行监督训练，使模型适应特定任务或领域。

**流程：**
1. 准备（指令, 回复）格式的标注数据集
2. 选择基础模型（LLaMA、Qwen、ChatGLM 等）
3. 配置训练超参（学习率通常 1e-5 ~ 5e-5，比预训练小 1-2 个数量级）
4. 训练并评估（loss、BLEU、人工评测）

**适用场景：** ChatBot、垂直行业助手、指令跟随能力提升

### 4.3 LoRA（Low-Rank Adaptation）

**核心原理：**
```
W' = W + ΔW = W + BA
```
- W 为预训练参数（冻结），B∈Rᵈˣʳ，A∈Rʳˣᵏ（r 远小于 d, k）
- 只训练低秩矩阵 A 和 B，参数量约为原来的 1-3%
- 推理时合并 `W + BA`，无额外延迟

**常用超参：**
- `r`：秩，通常 4~64，越大效果越接近全量微调
- `lora_alpha`：缩放系数，通常等于 r 或 2r
- `target_modules`：通常注入 `q_proj`, `v_proj`，也可加 `k_proj`, `o_proj`

**优势：** 显存需求低、支持多 adapter 灵活切换、效果接近全量微调

### 4.4 QLoRA（Quantized LoRA）

**核心思想：** LoRA + 4bit NormalFloat 量化

- 将基础模型量化到 4bit（NF4 格式），显存减少约 4 倍
- 在量化模型上应用 LoRA 进行微调
- 典型场景：RTX 4090（24GB）可微调 70B 参数模型

**技术细节：**
- 引入 **Double Quantization**（二次量化）：量化 NF4 量化常数，进一步节省显存
- **Paged Optimizer**：用 CPU 内存 offload optimizer states

### 4.5 RLHF（基于人类反馈的强化学习）

**三阶段流程：**

```
阶段 1: SFT 阶段
  └── 用高质量示范数据微调预训练模型

阶段 2: 奖励模型训练
  └── 收集人类对多个回答的偏好排序数据
      训练 Reward Model（RM）预测人类偏好

阶段 3: RL 优化
  └── 用 PPO 算法，以 RM 分数为奖励信号
      优化 SFT 模型，使输出更符合人类期望
      （同时用 KL 散度约束，防止模型偏离太远）
```

**应用：** ChatGPT、InstructGPT、Claude 等商业模型的对齐核心技术

### 4.6 DPO（直接偏好优化）

**核心思想：** 绕过奖励模型，直接用偏好数据优化语言模型

**对比 RLHF：**
| | RLHF | DPO |
|--|------|-----|
| 奖励模型 | 需要单独训练 | 不需要 |
| 训练复杂度 | 高（PPO 不稳定） | 低（标准分类损失） |
| 效果 | 强 | 接近 RLHF |
| 应用趋势 | 老牌方法 | 越来越主流 |

**数据格式：**（prompt, chosen_response, rejected_response）

### 4.7 微调选型建议

```
场景判断：
├── 显存 < 24GB    → QLoRA
├── 显存 24-80GB   → LoRA
├── 多机多卡        → 全量微调（DeepSpeed/FSDP）
├── 快速验证效果    → LoRA (r=8 ~ 16)
└── 生产部署精度要求高 → 全量微调
```

## 五、RAG 检索增强生成

### 5.1 RAG 原理与动机

**RAG（Retrieval-Augmented Generation）** 将外部知识检索与 LLM 生成相结合：

```
用户问题
    │
    ▼
[向量化 Query]
    │
    ▼
[向量数据库检索 Top-K 文档块]
    │
    ▼
[拼装 Prompt: 问题 + 检索到的上下文]
    │
    ▼
[LLM 生成回答]
```

**RAG vs 微调对比：**

| 维度 | RAG | 微调（Fine-tuning） |
|-----|-----|-------------------|
| 知识更新 | 实时，直接更新知识库 | 需重新训练 |
| 训练成本 | 无需训练 | 较高 |
| 知识边界 | 受检索质量影响 | 融入模型参数 |
| 幻觉控制 | 有依据可追溯 | 较难控制 |
| 适用场景 | 时效性强、知识量大 | 风格/指令/领域能力迁移 |

### 5.2 完整 RAG 流程

**Step 1: 数据收集与加载**
- 支持格式：PDF、Word、HTML、Markdown、数据库
- 工具：LangChain Loaders、LlamaIndex Readers、Unstructured

**Step 2: 文档清洗**
- OCR 文本纠错、标点统一、噪声去除
- 结构化标注（标题/正文/代码/表格）

**Step 3: 文档分块（Chunking）**

| 策略 | 描述 | 适用场景 |
|-----|------|---------|
| 固定长度 | 按 token/字符数切割 | 纯文本、快速原型 |
| 语义边界 | 按段落/标题/句子切割 | 结构化文档 |
| 滑动窗口 | 相邻块 15-30% 重叠 | 长上下文依赖 |
| 递归分割 | 先按段落，再按句子，再按字符 | 通用场景（推荐） |

**常用参数：** chunk_size=500~1000 字符，chunk_overlap=50~200 字符

**Step 4: 向量化（Embedding）**
- 将文本块转为高维稠密向量
- 常用模型：BGE、text-embedding-ada-002、Sentence-BERT
- 中文推荐：BAAI/bge-m3、text2vec-large-chinese

**Step 5: 向量存储**

| 方案 | 特点 | 场景 |
|-----|------|-----|
| FAISS | 本地高性能，不支持持久化 CRUD | 原型、研究 |
| Milvus | 分布式，亿级，完整 CRUD | 企业生产环境 |
| Qdrant | 高性能 API，易部署 | Web 服务 |
| Weaviate | 多模态，RESTful | 多模态需求 |
| pgvector | PostgreSQL 插件，SQL 友好 | 轻量混合检索 |
| Elasticsearch | 全文+向量混合 | 关键词+语义混合 |

**Step 6: 检索**
- **稠密检索**：向量相似度（余弦/内积）
- **稀疏检索**：BM25 关键词匹配
- **混合检索**（推荐）：融合两者结果，提升覆盖率
- **重排序（Rerank）**：用 CrossEncoder 精排 Top-K 结果

**Step 7: 生成**
```python
prompt = f"""
你是一个专业助手。请根据以下参考内容回答用户的问题。
如果参考内容中没有相关信息，请如实告知。

参考内容：
{retrieved_context}

用户问题：{user_query}

请给出详细、准确的回答：
"""
response = llm.invoke(prompt)
```

### 5.3 RAG 优化策略

**检索优化：**
- Query 改写/扩展：将用户问题改写为多个子查询
- Hypothetical Document Embeddings (HyDE)：先让 LLM 生成假想答案，用答案向量检索
- 父子文档检索：小块索引 + 大块返回，兼顾精度与上下文

**生成优化：**
- 引用溯源：在回答中标注来源文档
- 置信度过滤：低相关度文档不纳入上下文
- Self-RAG：让模型自主决定是否需要检索

### 5.4 RAG 评估指标

| 评估维度 | 指标 | 说明 |
|---------|------|------|
| 检索质量 | Recall@K, Precision@K, NDCG | 检索到正确文档的比例 |
| 生成质量 | BLEU, ROUGE, BERTScore | 与参考答案的相似度 |
| 忠实度 | Faithfulness | 回答是否基于检索内容 |
| 相关性 | Answer Relevance | 回答是否回应了问题 |
| 落地评估 | 人工评测 | 实际业务准确率 |

**推荐框架：** RAGAS、TruLens

## 六、Agent 智能体开发

### 6.1 Agent 核心概念

**Agent = 感知 + 规划 + 行动 + 记忆**

```
用户输入
    │
    ▼
┌─────────────────────────────────┐
│             Agent 核心           │
│  思考（Thought）                │
│  行动（Action）→ 调用工具        │
│  观察（Observation）← 工具返回  │
│  （循环直到任务完成）            │
└─────────────────────────────────┘
    │
    ▼
最终回答（Final Answer）
```

### 6.2 ReAct 框架

**ReAct = Reasoning + Acting（推理 + 行动）**

经典 Prompt 格式：
```
Question: 今天北京的天气怎么样？最高温度是多少摄氏度？

Thought: 我需要查询今天北京的天气信息
Action: get_weather
Action Input: {"city": "北京", "date": "today"}
Observation: 北京今天晴，最高气温 28°C，最低气温 15°C

Thought: 我已经得到了所需信息
Final Answer: 今天北京天气晴，最高温度为 28 摄氏度
```

**优势：**
- 可解释性：每步推理过程透明
- 减少幻觉：基于工具返回的真实数据
- 支持复杂任务拆解

### 6.3 工具调用（Function Call）

现代 LLM API 支持 Function Call，让模型决定何时调用哪个工具：

```python
tools = [
    {
        "name": "search_web",
        "description": "搜索互联网获取最新信息",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "搜索关键词"}
            },
            "required": ["query"]
        }
    }
]

response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "最新的大模型发展动态？"}],
    tools=tools,
    tool_choice="auto"
)
```

**MCP（Model Context Protocol）：** Anthropic 提出的标准化工具调用协议，让 Agent 与外部系统解耦通信。

### 6.4 Agent 记忆系统

```
短期记忆（Short-term Memory）
  └── 当前对话上下文（In-context Window）
      自动压缩摘要（超长时截断/压缩）

长期记忆（Long-term Memory）
  ├── 向量数据库（偏好、历史事件）
  ├── 图数据库（实体关系网络）
  └── 关系数据库（结构化用户画像）
```

### 6.5 多 Agent 架构

**协作模式：**
- **顺序管道**：Agent A → Agent B → Agent C
- **并行分工**：多个 Agent 同时处理不同子任务
- **监督者模式**：Supervisor Agent 分配任务给 Worker Agents
- **反思机制**：Critic Agent 评审 Actor Agent 的输出

**主流框架：**
- LangGraph：节点+边的有向图，适合复杂状态管理
- AutoGen：微软推出，擅长多 Agent 对话协作
- CrewAI：角色扮演式多 Agent 框架

### 6.6 Agent 系统设计注意点

1. **工具稳定性**：工具接口异常要有重试和降级策略
2. **推理漂移**：设置最大循环次数，防止无限循环
3. **Cost 控制**：每次 LLM 调用都有 token 消耗，需合理规划
4. **可观测性**：记录每步 Thought/Action/Observation，便于调试
5. **安全性**：沙箱执行代码工具，审计高权限操作

## 七、Prompt 工程

### 7.1 提示类型对比

| 类型 | 定义 | 适用场景 |
|-----|------|---------|
| Zero-shot | 直接描述任务，无示例 | 简单通用任务 |
| Few-shot | 提供 2-5 个示例 | 格式要求明确的任务 |
| Chain-of-Thought (CoT) | 引导模型逐步推理 | 复杂数学/逻辑问题 |
| Tree-of-Thought (ToT) | 分支探索多条推理路径 | 需要回溯的复杂问题 |
| ReAct | 推理+工具调用融合 | 需要外部信息的任务 |
| Self-Consistency | 多次采样取多数投票 | 提升推理准确性 |

### 7.2 CoT 示例

```
# 普通提示（效果差）
Q: 小明有 3 包糖，每包 6 颗，他吃了 8 颗，还剩多少？
A: 10 颗

# CoT 提示（效果好）
Q: 小明有 3 包糖，每包 6 颗，他吃了 8 颗，还剩多少？
A: 第一步，计算总糖数：3 × 6 = 18 颗
   第二步，减去吃掉的：18 - 8 = 10 颗
   所以还剩 10 颗
```

**Zero-shot CoT 魔法咒语：** "Let's think step by step" / "让我们一步一步思考"

### 7.3 Prompt 优化技巧

1. **角色设定**：`你是一名专业的xxx，擅长xxx`
2. **明确输出格式**：`请以 JSON 格式输出，包含字段：title, summary, tags`
3. **添加约束**：`回答须在 200 字以内，不得包含主观判断`
4. **提供上下文**：相关背景信息放在提示前半段
5. **分隔符**：用 `---` 或 `###` 分隔不同部分，防止注入
6. **Prompt Caching**：将长而固定的系统提示缓存，减少 token 消耗（Claude/GPT 支持）

### 7.4 采样参数

| 参数 | 含义 | 推荐值 |
|-----|------|-------|
| temperature | 控制输出随机性（0=确定性，2=高随机） | 创意任务: 0.7-1.0；问答: 0.0-0.3 |
| top_p | 核采样，累积概率阈值 | 0.9（与 temperature 配合使用） |
| top_k | 每步只从概率最高 k 个 token 采样 | 40-50 |

| stop | 停止生成的标记 | `["\n\n", "###"]` |

## 八、推理加速与部署

### 8.1 KV Cache

**原理：** 自回归生成时，新 token 依赖所有历史 K/V。KV Cache 将计算过的 K/V 缓存下来，避免重复计算。

**显存计算公式：**
```
KV Cache 大小 = 2 × batch_size × seq_len × num_layers × num_heads × head_dim × dtype_bytes
```

**优化方向：**
- **MQA（Multi-Query Attention）**：多个 Query 头共享一个 K/V，显著减少 KV 缓存
- **GQA（Grouped-Query Attention）**：MQA 和 MHA 的折中（LLaMA 2 70B 使用）
- **Paged KV Cache（vLLM）**：将 KV Cache 分页管理，减少显存碎片

### 8.2 FlashAttention

**核心问题：** 标准 Attention 的中间矩阵（N×N）存在 GPU HBM（慢速显存），IO 成为瓶颈

**FlashAttention 解法：**
- **分块计算（Tiling）**：将 Q/K/V 分小块，在 SRAM（快速缓存）中完成运算
- **流式 softmax**：在线计算 softmax，无需全局存储完整注意力矩阵
- **重计算**：反向传播时重新计算中间值，用算力换显存

**效果：** 显存复杂度 O(N²) → O(N)，训练/推理速度提升 2-4 倍

| 版本 | 特点 |
|-----|------|
| FlashAttention 1 | 基础版本 |
| FlashAttention 2 | 更高并行度，速度再提升 2x |
| FlashAttention 3 | 针对 Hopper/H100 架构优化 |

### 8.3 模型量化

**原理：** 将模型参数从高精度（FP32/FP16）降低到低精度（INT8/INT4/FP8）

**主流方案：**

| 方案 | 类型 | 精度 | 速度 | 适用 |
|-----|------|------|------|------|
| GPTQ | 训练后量化 (PTQ) | 好 | 快 | GPU 推理（4bit） |
| AWQ | 训练后量化 (PTQ) | 更好 | 快 | GPU 推理（4bit，感知激活） |
| INT8 (bitsandbytes) | 动态量化 | 较好 | 中 | 轻量部署 |
| GGUF (llama.cpp) | 混合精度 | 可调 | CPU 友好 | 本地部署 |

### 8.4 vLLM

**核心特性：**
- **PagedAttention**：将 KV Cache 分成物理页（类似操作系统虚拟内存），动态分配，减少碎片
- **连续批处理（Continuous Batching）**：动态将新请求加入批次，GPU 利用率极大提升
- **并行采样**：同一请求多个输出共享 prefix 的 KV Cache

**对比：**

| 方案 | 特点 | 适用场景 |
|-----|------|---------|
| vLLM | 高吞吐，低延迟 | 生产部署，高并发 |
| TensorRT-LLM | 英伟达优化，极致性能 | NVIDIA 专属硬件 |
| Ollama | 易用，本地部署 | 开发调试，个人使用 |
| HuggingFace | 生态丰富，灵活 | 研究，多模型切换 |

### 8.5 解码策略

| 策略 | 原理 | 特点 |
|-----|------|------|
| Greedy | 每步选概率最大 token | 确定性，可能陷入重复 |
| Beam Search | 维护 k 条候选路径 | 质量高，计算开销大 |
| Top-K | 从概率最高 K 个中随机采样 | 多样性好 |
| Top-P (Nucleus) | 从累积概率达 P 的最小集合中采样 | 动态平衡质量与多样性 |
| Temperature | 调整分布尖锐度 | 控制创意程度 |

## 十、高频八股题库（含答案）

### 10.1 基础理论类

**Q1: 请简述 Transformer 的整体架构**

> **答：** Transformer 由多层 Encoder 和 Decoder 堆叠组成。Encoder 层包含：多头自注意力（捕捉序列内部关系）+ FFN（非线性变换）+ 残差连接 + LayerNorm。Decoder 在此基础上增加对 Encoder 输出的交叉注意力。现代 LLM 多用 Decoder-Only（如 GPT 系列），仅使用因果（单向）自注意力。

---

**Q2: 自注意力机制的计算复杂度是多少？有什么缺点？**

> **答：** 时间复杂度和空间复杂度均为 O(n²d)，n 为序列长度，d 为维度。缺点：
> 1. 长序列显存暴涨（n=4096 时中间矩阵 16M 个元素）
> 2. 计算资源随序列长度二次增长
> 解决方案：FlashAttention（IO 优化）、线性注意力（Mamba/SSM）、稀疏注意力（Longformer）

---

**Q3: 为什么 Transformer 需要位置编码？RoPE 和 Sinusoidal 有何区别？**

> **答：** Transformer 中 Self-Attention 的计算是置换不变的（permutation-invariant），不具备位置感知能力，需显式注入位置信息。
> 
> - **Sinusoidal**：固定公式计算，无需学习参数，理论上可外推到任意长度，但实际外推效果有限
> - **RoPE（旋转位置编码）**：对 Q/K 向量施加旋转变换，Q·K 的内积天然编码相对位置，外推能力更强，是 LLaMA、Qwen 等主流模型的选择

---

**Q4: LayerNorm 和 BatchNorm 的区别？为什么 Transformer 用 LayerNorm？**

> **答：** 
> - **BatchNorm**：对 batch 中所有样本的同一特征维度归一化（需要足够 batch size，不适合序列变长）
> - **LayerNorm**：对单个样本的所有特征维度归一化（不依赖 batch size，适合序列模型）
> 
> Transformer 用 LayerNorm 的原因：NLP 任务序列长度不固定，BatchNorm 效果不稳定；LayerNorm 对 batch size 不敏感，适合并行训练。

---

**Q5: GPT 和 BERT 的核心区别是什么？**

> **答：** 
> - **GPT（Decoder-Only）**：自回归语言模型，只能看到左侧 token（因果掩码），适合生成任务（对话、补全）
> - **BERT（Encoder-Only）**：双向语言模型，通过 MLM（Masked Language Modeling）预训练，适合理解任务（分类、NER、QA）
> 
> 现代 LLM 主流架构（GPT-4、LLaMA、Qwen）均为 GPT 式 Decoder-Only。

---

### 10.2 微调技术类

**Q6: LoRA 的原理是什么？为什么比全量微调省显存？**

> **答：** LoRA 在冻结的预训练权重旁注入低秩矩阵：W' = W + BA（B∈R^{d×r}, A∈R^{r×k}，r≪d,k）。
> 
> 省显存原因：
> 1. 只有 A、B 参与梯度计算，参数量约为原来 1-3%
> 2. 冻结的 W 不需要存储梯度和 optimizer states（Adam 中 optimizer states 是参数量的 2 倍）
> 3. QLoRA 进一步将 W 量化为 4bit

---

**Q7: RLHF 和 DPO 的区别？什么情况下选择 DPO？**

> **答：**
> - **RLHF**：三阶段（SFT → RM 训练 → PPO 强化学习），需要单独训练奖励模型，PPO 训练复杂不稳定
> - **DPO**：直接用偏好数据 (chosen, rejected) 优化 LLM，无需奖励模型，损失函数简洁
> 
> **选择 DPO 的场景：** 计算资源有限、工程实现简单优先、效果接近 RLHF 即可接受时。当任务需要精细对齐且资源充足时，RLHF 效果更强。

---

**Q8: 微调时遇到 loss 为 NaN 或训练不稳定，如何排查？**

> **答：**
> 1. **降低学习率**：LLM 微调学习率通常 1e-5 ~ 5e-5
> 2. **梯度裁剪**：设置 `max_grad_norm=1.0`
> 3. **混合精度问题**：检查 bf16 vs fp16（fp16 易溢出，推荐 bf16）
> 4. **数据问题**：检查是否有空字符串、超长 token、特殊字符
> 5. **batch size 过小**：配合梯度累积增大等效 batch size

---

### 10.3 RAG 类

**Q9: 请设计一个完整的 RAG 系统**

> **答：** 设计思路分四个模块：
> 
> **1. 离线索引管道：**
> 数据采集 → 清洗 → 分块（RecursiveCharacterTextSplitter，500 token，100 重叠）→ BGE Embedding → 存入 Milvus
> 
> **2. 在线检索：**
> Query → 同一 Embedding 模型向量化 → 混合检索（向量 + BM25）→ CrossEncoder 重排序 → 取 Top-5 块
> 
> **3. 生成：**
> 拼装 Prompt（系统提示 + 检索结果 + 用户问题）→ LLM 生成 → 带引用来源
> 
> **4. 评估：**
> 用 RAGAS 框架评估 Faithfulness、Answer Relevance、Context Recall

---

**Q10: RAG 效果不好，有哪些优化方向？**

> **答：**
> 
> **检索层优化：**
> - 混合检索（向量 + 关键词），弥补各自不足
> - 重排序（Reranker）提升精度
> - 查询改写：扩展用户意图，多角度检索
> - 调整 chunk_size，过大漏信息，过小引入噪声
> 
> **生成层优化：**
> - Prompt 中明确指示"基于参考内容"
> - 低置信度时拒绝回答
> - 引用标注，方便核查
> 
> **数据层优化：**
> - 提升文档质量（去噪、结构化）
> - 层级索引：摘要节点 + 详细节点
> - 定期更新知识库

---

**Q11: RAG 和微调如何权衡选择？**

> **答：**
> 
> | 场景 | 推荐方案 |
> |-----|---------|
> | 知识频繁更新 | RAG |
> | 领域专有术语、风格迁移 | 微调（SFT） |
> | 知识库超大（TB 级） | RAG |
> | 需要特定格式输出 | SFT 或 Prompt 工程 |
> | 对话自然性/礼仪 | RLHF/DPO |
> | 综合最优 | RAG + 微调结合 |

### 10.4 Agent 类

**Q12: 解释 ReAct 框架，它和 CoT 有何不同？**

> **答：**
> - **CoT（Chain-of-Thought）**：引导模型在生成答案前输出推理步骤，是纯语言模型内部的推理，不调用外部工具
> - **ReAct**：推理 + 行动的融合，模型不仅思考，还可以调用外部工具（搜索、计算、API 等）并观察结果，动态调整下一步
> 
> ReAct 解决了 CoT 无法获取实时信息、无法执行实际操作的局限。

---

**Q13: 多 Agent 系统中如何防止 Agent 之间的信息不一致？**

> **答：**
> 1. **中央状态管理**：LangGraph 用共享 State 对象，所有 Agent 读写同一状态
> 2. **消息队列**：Agent 间通过消息队列异步通信，保证顺序性
> 3. **事务机制**：关键状态更新用事务保证原子性
> 4. **版本控制**：状态变更带时间戳和版本号，冲突时用最新版
> 5. **Supervisor 模式**：主控 Agent 协调资源，避免并发写冲突

---

### 10.5 推理加速类

**Q14: KV Cache 是什么？如何减少它的显存占用？**

> **答：** KV Cache 将自回归推理中每步计算的 K、V 向量缓存下来，下一步直接复用，避免重复计算。
> 
> 减少 KV Cache 显存：
> 1. **MQA/GQA**：多个 Q 头共享 K/V（减少 h 倍）
> 2. **KV Cache 量化**：FP16 → INT8/FP8（减少 2-4 倍）
> 3. **Paged KV Cache（vLLM）**：分页管理减少碎片
> 4. **Sliding Window Attention**：只保留最近 N 个 token 的 KV
> 5. **KV Cache 压缩**：StreamingLLM 等方法保留 attention sink

---

**Q15: FlashAttention 解决了什么问题？核心思路是什么？**

> **答：** 解决了标准 Attention 计算时 GPU HBM IO 成为瓶颈的问题（O(N²) 中间矩阵读写）。
> 
> 核心思路：
> 1. **分块（Tiling）**：将 Q/K/V 分成小块，在 GPU SRAM（片上高速缓存）内完成局部计算
> 2. **在线 softmax（Online Softmax）**：数值稳定的流式 softmax，无需存储完整 N×N 矩阵
> 3. **重计算（Recomputation）**：反向传播时重新计算中间值，牺牲算力换显存

### 10.6 系统设计类

**Q16: 如何设计一个高并发的 LLM 推理服务？**

> **答：**
> 
> **架构层面：**
> - 使用 vLLM 的 Continuous Batching，最大化 GPU 利用率
> - 部署多个推理副本，负载均衡（如 Nginx/K8s HPA）
> - 流式输出（Streaming）降低首 token 等待感
> 
> **优化层面：**
> - Prompt Caching：缓存相同系统提示的 KV Cache
> - 请求队列：防止峰值压垮服务
> - 模型量化（AWQ/GPTQ）：降低显存，同卡部署更大模型
> 
> **监控层面：**
> - 监控 TTFT（首 token 时间）、TPS（吞吐量）、GPU 利用率
> - 异常 token 过滤，超时熔断

---

**Q17: 请介绍 LangChain 中 Chain 和 Agent 的区别**

> **答：**
> - **Chain**：预定义的固定工作流，步骤顺序确定，适合流程固定的任务（如 RAG QA Pipeline）
> - **Agent**：让 LLM 动态决定下一步行动（选择工具、决定何时结束），适合步骤不确定、需要规划的复杂任务
> 
> 选择原则：流程可预测用 Chain（稳定、可预期）；需要自主决策用 Agent（灵活、但 token 消耗更多、不稳定性更高）

### 10.7 手撕代码类

**Q18: 手写 Scaled Dot-Product Attention（Python/PyTorch）**

```python
import torch
import torch.nn.functional as F

def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    Q: (batch, heads, seq_len, d_k)
    K: (batch, heads, seq_len, d_k)
    V: (batch, heads, seq_len, d_v)
    mask: (batch, 1, 1, seq_len) 或 (batch, 1, seq_len, seq_len)
    """
    d_k = Q.size(-1)
    
    # 计算注意力分数
    scores = torch.matmul(Q, K.transpose(-2, -1)) / (d_k ** 0.5)
    # scores: (batch, heads, seq_len, seq_len)
    
    # 应用掩码（因果掩码或 padding 掩码）
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))
    
    # softmax 归一化
    attn_weights = F.softmax(scores, dim=-1)
    
    # 加权聚合 Value
    output = torch.matmul(attn_weights, V)
    
    return output, attn_weights
```

**Q19: 手写 BPE（字节对编码）Tokenization 核心逻辑**

```python
from collections import Counter

def get_vocab(corpus):
    """将语料转为词频字典，字符间加空格"""
    vocab = Counter()
    for word in corpus:
        vocab[' '.join(list(word)) + ' </w>'] += 1
    return vocab

def get_pairs(vocab):
    """统计所有相邻字符对的频次"""
    pairs = Counter()
    for word, freq in vocab.items():
        symbols = word.split()
        for i in range(len(symbols) - 1):
            pairs[(symbols[i], symbols[i+1])] += freq
    return pairs

def merge_vocab(pair, vocab):
    """合并最高频的字符对"""
    new_vocab = {}
    bigram = ' '.join(pair)
    replacement = ''.join(pair)
    for word in vocab:
        new_word = word.replace(bigram, replacement)
        new_vocab[new_word] = vocab[word]
    return new_vocab

# 主循环：执行 num_merges 次合并
def bpe_train(corpus, num_merges=100):
    vocab = get_vocab(corpus)
    for _ in range(num_merges):
        pairs = get_pairs(vocab)
        if not pairs:
            break
        best_pair = max(pairs, key=pairs.get)
        vocab = merge_vocab(best_pair, vocab)
    return vocab
```
